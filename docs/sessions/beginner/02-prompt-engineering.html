What is a Prompt?

A prompt is the text you send to an LLM to get a response. But a great prompt isn't just a question — it's a carefully structured piece of communication that gives the model everything it needs to produce exactly what you want.

Think of it like giving instructions to a brilliant but very literal assistant who has read everything ever written but has zero context about you, your project, or your standards. The more precise and complete your instructions, the better the output.

The Anatomy of a Great Prompt

Prompt Anatomy
ROLE
You are a senior Python developer...
Sets the model's persona, expertise level, and communication style. Calibrates tone and vocabulary.
INSTRUCTION
Review the following code and identify any bugs...
The core task. Be specific about the verb: explain, summarize, rewrite, list, classify, translate, evaluate.
CONTEXT
This is a payment processing function used in a high-traffic e-commerce API...
Background the model needs to give relevant, targeted output. What's the use case? Who's the audience?
EXAMPLES
For example, a bug would be: "line 14 has an off-by-one error..."
Few-shot examples that demonstrate the exact style, format, and level of detail you want. Often the single biggest quality lever.
CONSTRAINTS
Focus only on security and correctness issues, not style. Do not suggest refactors...
What the model should NOT do. Scope guards that prevent irrelevant output and keep responses focused.
OUTPUT FORMAT
Respond with a numbered list. For each bug: line number, description, severity (Low/Med/High), fix.
Exactly how you want the response structured. Markdown, JSON, bullet points, table, code block — be explicit.
Not All Elements Are Always Needed

For simple tasks ("What's the capital of France?") you need none of these. For complex tasks, each element you add narrows the output space toward what you actually want. Start simple and add elements only when outputs are missing the mark.

The Bad vs Good Prompt Rewrite

The fastest way to understand prompt engineering is to see concrete before/after examples. These three rewrites demonstrate the same principles in different contexts.

Example 1: Email Writing

Vague Prompt
bad-email.txt
Write an email to my client about
the project delay.

Result: Generic, wrong tone, may be too apologetic or too casual, wrong length.

Precise Prompt
good-email.txt
You are a professional account manager
at a software consultancy.

Write a client email about a 2-week
delay on their mobile app launch.

Context: The delay is due to
unexpected API integration issues.
The client (TechCorp CEO) values
directness and technical detail.

Constraints:
- Under 150 words
- Acknowledge impact, explain cause,
  state new timeline, offer call
- Professional but not overly formal

Format: Plain email with subject line.

Example 2: Code Explanation

Vague Prompt
bad-code.txt
Explain this Python code to me.

[code here]

Result: May be too technical or too simple, unclear what level of explanation is wanted.

Precise Prompt
good-code.txt
You are a patient coding teacher
explaining to a complete beginner
(no CS background).

Explain what this Python function does:

[code here]

Structure your explanation as:
1. What problem does this solve? (1-2
   sentences, plain English)
2. Walk through line by line
3. Analogy: relate it to something
   from everyday life
4. One thing to watch out for

Avoid jargon. If you must use a
technical term, define it immediately.

Example 3: Summarization

Vague Prompt
bad-summary.txt
Summarize this article.

[article text]

Result: Unknown length, unknown focus, unclear format, may miss what you care about.

Precise Prompt
good-summary.txt
You are an analyst summarizing
research for a busy VP of Engineering.

Summarize the article below. Focus on:
- Technical findings and data points
- Business/cost implications
- Any risks or caveats mentioned

Format:
- TL;DR: 2 sentences max
- Key Points: 3-5 bullets
- Action Items: what should eng team do?

[article text]
The Core Pattern

Every rewrite follows the same pattern: Who is the model? + What exactly should it do? + For whom / in what context? + What format should it use? + What should it avoid? The more precisely you answer these five questions, the tighter your output quality range.

Prompting Techniques

Beyond the anatomy of a single prompt, there are proven techniques for structuring how you communicate with an LLM. Each has specific strengths and appropriate use cases.

Zero-Shot Prompting

Give the task with no examples. The model relies entirely on its training to understand what you want. Works well for straightforward, well-defined tasks.

zero-shot.txt
Classify the sentiment of this review as Positive, Negative, or Neutral.

Review: "The delivery was two days late and the packaging was damaged,
but the product itself works perfectly."

Few-Shot Prompting

Provide 2-5 examples of the exact input/output pattern you want before giving the actual task. The model infers the pattern from examples rather than from abstract description alone. This is extremely powerful for custom formats, domain-specific output, or unusual tasks.

few-shot.txt
Convert these customer messages into structured support tickets.

Message: "My login stopped working after the update yesterday"
Ticket: { "category": "Auth", "priority": "High", "summary":
  "Login broken post-update", "action": "Investigate auth service" }

Message: "Can I change my billing address?"
Ticket: { "category": "Billing", "priority": "Low", "summary":
  "Billing address update request", "action": "Direct to account settings" }

Message: "The app crashes every time I try to upload a file over 10MB"

Chain-of-Thought (CoT) Prompting

For complex reasoning, math, or logic tasks, explicitly ask the model to think step-by-step before giving its final answer. This dramatically improves accuracy on multi-step problems because it forces the model to decompose the problem rather than jumping to a conclusion.

chain-of-thought.txt
A store sells apples for $0.50 each and oranges for $0.75 each.
Sarah buys 4 apples and 3 oranges, pays with a $10 bill.
How much change does she receive?

Think through this step by step before giving your final answer.

The key phrase is "think step by step" (or "let's work through this," "show your reasoning"). This single addition can raise accuracy on math and logic problems from 40% to 90%+ in some benchmarks.

When to Use Each

Zero-shot for simple, well-understood tasks. Few-shot when you need specific formatting or domain-specific output. Chain-of-Thought for any problem involving math, logic, multi-step reasoning, or complex analysis. The techniques can be combined: few-shot CoT (examples that show step-by-step reasoning) is among the most powerful approaches.

System Prompts & Roles

When you use an LLM via API, you typically have access to a system prompt — a special message that sets the model's overall behavior before any user interaction begins. Think of it as programming the model's personality and constraints for an entire conversation.

How the Chat API Works

The OpenAI-style API uses three message roles:

  • system: Persistent instructions that apply to the entire conversation. Only the developer sets this.
  • user: The end user's messages.
  • assistant: The model's previous responses (included for conversation history).

Example: Customer Service Bot

system-customer-service.txt
You are a helpful customer support agent for Acme Electronics.

Your personality: Friendly, patient, solution-focused. Never
dismissive or technical unless the customer is clearly technical.

Your capabilities:
- Answer product questions using the knowledge base provided
- Help troubleshoot common issues (see troubleshooting guide)
- Process return requests (ask for order number first)
- Escalate to human agents for billing disputes or complex issues

Your constraints:
- Never discuss competitor products
- Do not make promises about shipping timelines not in our policy
- If you don't know the answer, say so and offer to escalate
- Keep responses under 150 words unless detailed steps are needed

Example: Coding Assistant

system-coding-assistant.txt
You are a senior software engineer and coding assistant specializing
in Python and TypeScript.

When helping with code:
- Default to type annotations and modern syntax
- Prioritize readability over cleverness
- Always explain WHY, not just WHAT
- Point out potential bugs or edge cases proactively
- If the approach has better alternatives, mention them briefly

When reviewing code:
- Be direct about problems, not just "consider using X"
- Classify issues: bug / style / performance / security

Format code blocks with the appropriate language tag.
For complex explanations, use numbered steps.

Example: Teacher Persona

system-teacher.txt
You are a patient, encouraging teacher for adult learners studying AI
concepts for the first time. Your students have no technical background.

Teaching style:
- Use concrete analogies before abstract definitions
- Introduce vocabulary slowly; always define terms immediately
- Validate that questions are good questions (they are)
- After explaining, check understanding: "Does this make sense?
  What questions do you have?"
- Never say "it's simple" or "obviously" — nothing is obvious to a beginner

If a student seems frustrated, acknowledge it:
"This concept trips up a lot of people at first. Let's try a
different angle."
System Prompts Are Your API's Configuration

When you build an AI product, the system prompt is your primary tool for shaping user experience. It sets the model's tone, capabilities, constraints, and personality for every interaction. Well-crafted system prompts are the difference between a frustrating chatbot and one that feels genuinely helpful and in-brand.

Common Pitfalls

Even experienced developers fall into these traps. Recognizing them is the first step to avoiding them.

Pitfall 1: Ambiguous Instructions

The Problem

Vague words like "good," "professional," "brief," and "detailed" mean different things to different people — and to the model. "Write a brief summary" could mean 2 sentences or 2 paragraphs. "Make it more professional" — professional for a law firm or a startup?

Fix: Quantify and specify. "Under 50 words" not "brief." "Use formal language appropriate for a Fortune 500 board memo" not "professional."

Pitfall 2: Conflicting Constraints

The Problem

Instructions that contradict each other leave the model to guess which takes priority. "Be comprehensive AND concise" — which wins? "Explain it for a beginner BUT use proper technical terminology" — these can't both be fully satisfied.

Fix: When constraints conflict, make the priority explicit. "Be comprehensive — completeness over brevity here" or "explain it for a beginner and briefly define technical terms when they're necessary."

Pitfall 3: Overloading One Prompt

The Problem

Asking the model to do 5 different things in one prompt almost always produces worse results than breaking it into sequential prompts. "Analyze this document, summarize it, extract action items, identify risks, translate it to Spanish, and format it as a report" — the model tries to do everything at mediocre quality.

Fix: Chain prompts. Do the analysis first, get that output, then summarize it, then translate. Each step gets full model attention and you can verify quality before proceeding.

Pitfall 4: Forgetting the Negative Space

The Problem

Telling the model what you want is necessary but often insufficient. Without explicit "do NOT" instructions, the model will include things you didn't want: preamble ("Great question!"), caveats, disclaimers, suggestions you didn't ask for, or alternative interpretations of your task.

Fix: Add a constraints section. "Do not add preamble or caveats. Do not suggest alternatives unless asked. Return only the requested output."

Prompt Rewrite Workshop
Transform vague prompts into precise, high-quality instructions
  1. Start with the classic vague prompt
    Here's the original, unfocused prompt:
    vague.txt
    Write something about Python.
    This produces wildly inconsistent results. It could be a poem, a history essay, a tutorial — the model has no idea what you actually need.
  2. Apply the anatomy step by step
    Let's add each element:
    • Role: You are a Python tutor
    • Instruction: Explain what Python is and why beginners should learn it
    • Context: For a complete beginner with no programming background
    • Constraints: Under 100 words, no jargon (define any technical term you use)
    • Output format: Three short bullet points
  3. The improved prompt
    improved.txt
    You are a patient Python tutor teaching adults with zero
programming experience.

Explain what Python is and why it's a great first programming
language to learn.

Audience: Complete beginner, no technical background.
Constraints: Under 100 words total. No unexplained jargon —
if you use a technical term, define it in parentheses immediately.
Format: Exactly 3 bullet points.
    Notice how specific and constrained this is. The model now has almost no room to produce something you didn't want.

Brand voice: approachable, slightly playful, confidence-inspiring.

Write a product description for: [product name + 3-4 key features]

not technically savvy, values simplicity and reliability"]

Format:
- Headline: benefit-first, max 10 words
- Body: 2 short paragraphs, ~50 words each
- CTA: one sentence

What Makes an Agent?

A basic LLM call takes input and returns output — one step, no memory, no ability to act in the world. An AI agent is fundamentally different: it combines an LLM's reasoning with tools it can use, memory it can maintain, and a planning loop that iterates until the goal is achieved.

The four components of an agent:

Component What It Does Example
LLM (Core) Reasons about what to do next; decides which tools to call GPT-4o, Claude 3.5, Llama 3
Tools Functions the agent can call to affect the world or gather info Web search, code execution, email sending, database query
Memory Stores context across steps (short-term) or sessions (long-term) Conversation history, vector store of past interactions
Planning Loop Iterates: reason → act → observe → reason again until done ReAct loop, chain-of-thought, tree-of-thought
ReAct Loop:
THOUGHT: "I need to find today's weather in London to answer this."
ACTION: get_weather("London")
OBSERVATION: {"temp": 14, "condition": "Rainy"}
THOUGHT: "I now have the weather. I can answer."
FINAL ANSWER: "It's currently 14°C and rainy in London."
Core Concept

An agent is an LLM that can use tools, remember context, and take multi-step actions to achieve a goal. The critical difference from a simple chatbot is the loop — the agent keeps acting until it decides the task is complete.

Tool Calling

Tools are what give agents the ability to act in the world. Common tool categories include: web search, code execution, file operations, database queries, and external API calls. The agent selects tools based on their descriptions — clear, accurate descriptions are essential.

agent_tools.py
import os, json, math
from openai import OpenAI

client = OpenAI(api_key=os.environ["OPENAI_API_KEY"])

# ---- Define tools ----
tools = [
    {
        "type": "function",
        "function": {
            "name": "get_weather",
            "description": "Get current weather for a city. Use for weather/temperature questions.",
            "parameters": {
                "type": "object",
                "properties": {
                    "city": {"type": "string", "description": "City name"},
                },
                "required": ["city"],
            },
        },
    },
    {
        "type": "function",
        "function": {
            "name": "calculate",
            "description": "Evaluate a mathematical expression. Use for arithmetic, not language tasks.",
            "parameters": {
                "type": "object",
                "properties": {
                    "expression": {
                        "type": "string",
                        "description": "Python math expression, e.g. '2 ** 10' or 'math.sqrt(144)'"
                    },
                },
                "required": ["expression"],
            },
        },
    },
]

# ---- Tool implementations ----
def get_weather(city: str) -> dict:
    # In production: call a real weather API
    return {"city": city, "temperature": 18, "condition": "Sunny", "units": "celsius"}

def calculate(expression: str) -> dict:
    try:
        result = eval(expression, {"__builtins__": {}, "math": math})
        return {"result": result, "expression": expression}
    except Exception as e:
        return {"error": str(e)}

def execute_tool(name: str, arguments: str) -> str:
    args = json.loads(arguments)
    if name == "get_weather":
        return json.dumps(get_weather(**args))
    elif name == "calculate":
        return json.dumps(calculate(**args))
    return json.dumps({"error": "Unknown tool"})

# ---- Agent loop ----
def run_agent(user_query: str, max_steps: int = 5) -> str:
    messages = [{"role": "user", "content": user_query}]

    for step in range(max_steps):
        response = client.chat.completions.create(
            model="gpt-4o-mini",
            messages=messages,
            tools=tools,
        )
        choice = response.choices[0]
        messages.append(choice.message)

        if choice.finish_reason == "stop":
            return choice.message.content

        if choice.finish_reason == "tool_calls":
            for tool_call in choice.message.tool_calls:
                result = execute_tool(
                    tool_call.function.name,
                    tool_call.function.arguments
                )
                messages.append({
                    "role": "tool",
                    "tool_call_id": tool_call.id,
                    "content": result,
                })

    return "Max steps reached without a final answer."

print(run_agent("What's the weather in Paris, and what's 15% of the temperature in Fahrenheit?"))

Memory Systems

Memory determines what an agent knows at any point during execution. There are three distinct types, each serving a different purpose:

Memory Type Storage Lifespan Use Case
Short-term LLM messages array (in-context) Current session only Conversation context, multi-step task state
Long-term Vector database (retrieved by similarity) Persists across sessions User preferences, past interactions, knowledge base
Episodic Structured store (SQL or key-value) Persists; exact lookup "Last time you asked about X, here's what happened"

For most production agents, you'll combine short-term (conversation messages) with long-term vector memory. The agent retrieves relevant past context at the start of each turn to supplement what it can see in the conversation window.

Combining memory types

Most production agents use both conversation history AND vector-based long-term memory. At the start of each turn: (1) retrieve relevant long-term memories, (2) inject them as context, (3) run the conversation with the combined context. This simulates a much larger "effective context window."

Multi-Agent Systems

A single agent has a limited context window and can only do one thing at a time. Multi-agent systems solve this by decomposing a complex task into specialized sub-tasks, each handled by a purpose-built agent. An orchestrator coordinates the flow.

Example pipeline: Research → Write → Review

  • Research Agent: Given a topic, runs web searches and collects raw notes.
  • Writing Agent: Takes the research notes, writes a structured draft.
  • Review Agent: Critiques the draft for accuracy, tone, and completeness; outputs a revised version.
Framework Style Best For
AutoGen Conversational multi-agent; agents talk to each other Debate, iterative refinement, coding tasks
LangGraph Graph-based workflow; explicit state machine Complex branching workflows, production apps
CrewAI Role-based crews with tasks and goals Rapid prototyping; agent "crews" with defined roles

Agent Failure Modes

Agents fail in ways that simple LLM calls don't. Understanding the failure modes lets you build safeguards before they hit production.

Failure Mode What Happens Prevention
Infinite loops Agent keeps calling tools without making progress; costs spiral Hard limit on max iterations (e.g., 10 steps)
Hallucinated tool calls Agent invents tool names or arguments that don't exist Strict tool name validation; return clear errors on bad calls
Prompt injection in tool outputs A web search result contains "Ignore previous instructions…" Sanitize tool outputs; mark them clearly as untrusted data
Irreversible actions Agent sends an email, deletes data, or charges a credit card prematurely Human-in-the-loop checkpoints before irreversible steps
Always implement a step limit

Always implement a maximum iteration count — agents can loop indefinitely without it, consuming tokens and incurring costs. A limit of 10–20 steps is standard for most tasks. If the agent hasn't finished by then, it either needs to be redesigned or requires human escalation.

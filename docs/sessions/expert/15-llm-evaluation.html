LLM Evaluation

Expert 60 min Session 15 of 20

Why LLM Evaluation is Hard

With traditional ML classification, evaluating quality is straightforward: compare the predicted class to the true label, compute accuracy. LLMs break this model entirely. A question like "Explain quantum entanglement in simple terms" has no single correct answer — there are hundreds of equally valid, high-quality responses. And dozens of technically correct but useless ones.

The challenges stack up:

  • Open-ended outputs: No single ground truth. Multiple valid answers exist.
  • Context-dependence: "Good" depends on the user's background, the use case, the desired tone.
  • Hallucination is subtle: Incorrect information often sounds more confident and fluent than correct information.
  • Latent quality dimensions: Helpfulness, accuracy, safety, coherence, and tone are all important — and often trade off against each other.
Vibes don't scale

Vibes-based evaluation — "it seems pretty good when I try it" — doesn't scale. You need systematic evaluation to improve your AI product. Without it, you can't tell whether a prompt change made things better or worse, and you'll ship regressions without knowing.

Golden Datasets

A golden dataset is a curated collection of test cases with human-verified expected outputs. Each example has three parts: a question or input, a reference answer, and an evaluation rubric explaining what "good" looks like for this input.

Scale When to Use Build Time
20 examples Early prototype, sanity check, daily development feedback 1–2 hours
200 examples Pre-launch validation, prompt A/B testing with statistical confidence 1–2 days
2000 examples Production regression suite, fine-tuning evaluation, safety red-teaming 1–2 weeks

What makes a high-quality golden dataset:

  • Diverse coverage: Include easy, hard, edge cases, and adversarial inputs.
  • Real user queries: Don't make up examples — sample from actual user traffic when possible.
  • Clear rubrics: Annotators and the LLM judge must agree on what "correct" means.
  • Inter-annotator agreement: If two humans can't agree, the question is ambiguous — clarify or discard it.
Start with 50

Start with 50 diverse, high-quality examples. Quality beats quantity for early-stage evals. A 50-example dataset you trust is far more valuable than 500 examples with inconsistent or unclear expected answers.

Automated Evaluation

Automated metrics fall on a spectrum from simple (fast, cheap, brittle) to sophisticated (slow, expensive, accurate). For production evals, you'll typically combine several:

Method What It Measures Best For Limitations
String match / regex Exact output format compliance JSON structure, required keywords, forbidden words Zero tolerance for valid paraphrases
BLEU / ROUGE N-gram overlap with reference text Summarization (rough), machine translation Poor for open-ended generation; misses semantic equivalence
BERTScore Semantic similarity to reference using BERT embeddings Summarization quality with paraphrase tolerance Still requires a reference answer
LLM-as-judge Holistic quality scoring against a rubric Open-ended tasks, helpfulness, coherence, tone Slower, costs LLM API calls, judge bias
llm_as_judge.py
import os, json
from openai import OpenAI

client = OpenAI(api_key=os.environ["OPENAI_API_KEY"])

JUDGE_PROMPT = """You are an expert evaluator of AI assistant responses.
Score the following response on a scale of 1-5 for each dimension,
then provide an overall score.

## Evaluation Dimensions
- Accuracy (1-5): Is the information factually correct?
- Helpfulness (1-5): Does it actually answer the question?
- Clarity (1-5): Is it easy to understand?
- Conciseness (1-5): Is it appropriately concise (not padded)?

## Output Format (JSON only, no explanation outside the JSON)
{
  "accuracy": <1-5>,
  "helpfulness": <1-5>,
  "clarity": <1-5>,
  "conciseness": <1-5>,
  "overall": <1-5>,
  "rationale": ""
}"""

def judge_response(question: str, response: str) -> dict:
    prompt = f"QUESTION: {question}\n\nRESPONSE TO EVALUATE:\n{response}"
    result = client.chat.completions.create(
        model="gpt-4o",  # use a strong judge
        messages=[
            {"role": "system", "content": JUDGE_PROMPT},
            {"role": "user",   "content": prompt},
        ],
        response_format={"type": "json_object"},
    )
    return json.loads(result.choices[0].message.content)

# Example
question = "What is the capital of France?"
good_response = "The capital of France is Paris, which has been the country's capital since the 10th century."
bad_response  = "France is a country in Western Europe known for the Eiffel Tower and great food."

print("Good response score:", judge_response(question, good_response))
print("Bad response score: ", judge_response(question, bad_response))

Hallucination Detection

Hallucination detection in RAG systems focuses on two properties: faithfulness (does the answer contradict the retrieved context?) and groundedness (is every claim in the answer supported by the context?). An answer can be factually correct in the real world but still "unfaithful" to the wrong source documents.

faithfulness_eval.py
import os, json
from openai import OpenAI

client = OpenAI(api_key=os.environ["OPENAI_API_KEY"])

FAITHFULNESS_PROMPT = """You are evaluating whether an AI assistant's answer is faithful
to the provided source context. An answer is faithful if every claim it makes is
directly supported by (or at least not contradicted by) the context.

## Task
Given the CONTEXT and the ANSWER, identify any claims in the answer that are:
1. SUPPORTED — clearly backed by the context
2. NOT_IN_CONTEXT — not mentioned in the context (the model may have used training knowledge)
3. CONTRADICTED — the context says something different

Output JSON only:
{
  "faithfulness_score": <0.0 to 1.0>,
  "supported_claims": ["..."],
  "not_in_context_claims": ["..."],
  "contradicted_claims": ["..."],
  "verdict": "FAITHFUL" | "PARTIALLY_FAITHFUL" | "UNFAITHFUL"
}"""

def check_faithfulness(context: str, answer: str) -> dict:
    prompt = f"CONTEXT:\n{context}\n\nANSWER:\n{answer}"
    result = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[
            {"role": "system", "content": FAITHFULNESS_PROMPT},
            {"role": "user",   "content": prompt},
        ],
        response_format={"type": "json_object"},
    )
    return json.loads(result.choices[0].message.content)

# Test case
context = "Our refund policy allows returns within 30 days of purchase with a receipt."
faithful_answer  = "You can return items within 30 days if you have a receipt."
unfaithful_answer = "You can return items within 60 days, and no receipt is required."

print("Faithful:  ", check_faithfulness(context, faithful_answer)["verdict"])
print("Unfaithful:", check_faithfulness(context, unfaithful_answer)["verdict"])

Building an Eval Pipeline

An eval pipeline runs your golden dataset against your current prompt/model configuration and outputs scores for each test case. The key design principle: treat it like a unit test suite — run it on every change, track results over time, and fail the "deployment" if scores drop below a threshold.

eval_runner.py
import os, json, datetime
from openai import OpenAI

client = OpenAI(api_key=os.environ["OPENAI_API_KEY"])

# --- Test cases: your golden dataset ---
TEST_CASES = [
    {
        "id": "tc_001",
        "question": "What is our refund policy?",
        "expected_keywords": ["30 days", "receipt"],
        "context": "Refund policy: 30-day returns with receipt required.",
    },
    {
        "id": "tc_002",
        "question": "Do you offer free shipping?",
        "expected_keywords": ["free shipping", "$50"],
        "context": "Free shipping on orders over $50 for premium members.",
    },
]

SYSTEM_PROMPT_V1 = "You are a helpful support assistant. Answer based on the context."
SYSTEM_PROMPT_V2 = """You are a helpful support assistant.
Answer ONLY based on the context provided. Be concise (1-2 sentences).
If the answer is not in the context, say 'I don't have that information.'"""

def run_model(system: str, question: str, context: str) -> str:
    resp = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[
            {"role": "system", "content": system + f"\n\nCONTEXT:\n{context}"},
            {"role": "user",   "content": question},
        ]
    )
    return resp.choices[0].message.content

def keyword_score(response: str, keywords: list[str]) -> float:
    """Simple keyword-presence score (0 to 1)."""
    hits = sum(1 for kw in keywords if kw.lower() in response.lower())
    return hits / len(keywords)

def run_eval(prompt_name: str, system_prompt: str):
    scores = []
    for tc in TEST_CASES:
        response = run_model(system_prompt, tc["question"], tc["context"])
        score = keyword_score(response, tc["expected_keywords"])
        scores.append(score)
        print(f"  [{tc['id']}] score={score:.2f} | response: {response[:80]}...")

    avg = sum(scores) / len(scores)
    print(f"\n  {prompt_name} average score: {avg:.2f}\n")
    return avg

print("=== Eval Run:", datetime.datetime.now().isoformat(), "===\n")
v1_score = run_eval("PROMPT_V1", SYSTEM_PROMPT_V1)
v2_score = run_eval("PROMPT_V2", SYSTEM_PROMPT_V2)

winner = "V2" if v2_score > v1_score else "V1"
print(f"Winner: {winner} ({v2_score:.2f} vs {v1_score:.2f})")
Lab: Build an Eval Suite
Create 10 test cases, run them against two prompt versions, and compare scores

You'll build a complete evaluation suite for a FAQ bot, testing two prompt versions against 10 golden test cases. By the end, you'll have a quantitative comparison and know which prompt version is objectively better.

  1. Write 10 golden test cases
    Pick a topic (cooking, software support, travel booking). Write 10 question/context/expected-keywords tuples. Include 2-3 edge cases (out-of-scope questions, ambiguous queries).
  2. Design two prompt variants
    Write Prompt V1 (simple) and Prompt V2 (with explicit format instructions, context citation requirements, and a fallback instruction). Keep all other variables constant.
  3. Run keyword-based eval
    Use the eval_runner.py pattern to score both prompts on all 10 test cases. Record per-case scores and averages.
  4. Add LLM-as-judge scoring
    Use the judge_response() function to score each response on helpfulness and accuracy. Average these scores alongside keyword scores.
  5. Write a summary report
    Print a table comparing V1 vs V2 on every test case across both metrics. Identify where V2 improved and where it didn't. Form a hypothesis about why.

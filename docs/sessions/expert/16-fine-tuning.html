Fine-Tuning LLMs

Expert 75 min Session 16 of 20

The Decision: Prompt vs RAG vs Fine-tune

Fine-tuning is the most powerful option — and the most expensive, time-consuming, and risky. Before committing to it, work through this decision tree to make sure it's actually the right tool for your problem.

Decision Tree:
1. Can a well-crafted system prompt solve it? → Use prompting first.
2. Does the model need access to your private/up-to-date data? → Use RAG.
3. Is the output format or style extremely specific and the prompt is getting too long? → Consider fine-tuning.
4. Is the task very domain-specific and general models consistently fail? → Fine-tuning might help.
5. Is latency critical and your system prompt is huge? → Fine-tune to internalize instructions.
Approach Cost When It Wins Limitations
Prompting Lowest Most tasks; fast iteration; no training needed Long prompts cost more; inconsistent on complex formats
RAG Low–Medium Private/dynamic knowledge; verifiable facts Retrieval can fail; adds latency; requires doc pipeline
Fine-tuning High Consistent style/format; domain behavior; prompt compression Training cost; data prep burden; knowledge goes stale; risk of overfitting
The golden rule of fine-tuning

Fine-tuning is not a shortcut for giving the model knowledge — use RAG for that. Fine-tune for style, format, and behavior. Think of it as training a new employee: you use training to teach how to do the job, not to stuff their brain with a knowledge base.

How Fine-Tuning Works

Fine-tuning starts from a pre-trained model and continues gradient descent on your smaller, task-specific dataset. The model's weights are nudged to produce outputs closer to your training examples. The key challenge: doing this without destroying the broad capabilities the model already has (catastrophic forgetting).

Modern fine-tuning techniques avoid this by updating only a small fraction of weights:

Method Parameters Trained GPU Memory Use When
Full fine-tune All weights (7B–70B+) Very high (80GB+ for 13B) Maximum performance; you have GPU cluster access
LoRA Small rank matrices (~0.1% of weights) Low (runs on single A100 or even T4) Most production fine-tuning; best efficiency/quality tradeoff
RLHF Full weights + reward model Very high Alignment training; used by OpenAI, Anthropic, Meta for base models
DPO Full or LoRA weights Medium Preference-based alignment without RL; simpler than RLHF

How LoRA works: Instead of updating the full weight matrix W (which is enormous), LoRA adds two small matrices A and B where A × B approximates the weight update ΔW. If W is 4096 × 4096 = 16M parameters, a LoRA with rank 16 adds only 4096×16 + 16×4096 = 131K parameters — 120x fewer. Only A and B are trained; the original W is frozen.

Preparing Your Dataset

Dataset quality is the single biggest factor in fine-tuning success. The format differs by platform, but the quality criteria are universal: consistent format, correct outputs, diverse coverage.

Minimum dataset sizes:

  • 50–100 examples: Simple format/style tasks (OpenAI GPT-3.5 fine-tune)
  • 200–500 examples: Domain adaptation, complex formatting
  • 1000+ examples: Specialized behavior, code generation, classification
dataset_format.py
import json

# --- OpenAI fine-tuning format (JSONL) ---
# Each line is one training example with a complete conversation

openai_examples = [
    {
        "messages": [
            {"role": "system", "content": "You are a JSON extraction assistant. Output only valid JSON."},
            {"role": "user", "content": "Extract name and age: 'John Smith is 34 years old.'"},
            {"role": "assistant", "content": '{"name": "John Smith", "age": 34}'},
        ]
    },
    {
        "messages": [
            {"role": "system", "content": "You are a JSON extraction assistant. Output only valid JSON."},
            {"role": "user", "content": "Extract name and age: 'Maria Garcia, age 28, joined last week.'"},
            {"role": "assistant", "content": '{"name": "Maria Garcia", "age": 28}'},
        ]
    },
]

# Write to JSONL
with open("training_data.jsonl", "w") as f:
    for example in openai_examples:
        f.write(json.dumps(example) + "\n")

print(f"Wrote {len(openai_examples)} training examples")

# --- HuggingFace / Alpaca format (instruction fine-tuning) ---
hf_examples = [
    {
        "instruction": "Classify the sentiment of this review.",
        "input": "The product arrived broken and customer service was unhelpful.",
        "output": "NEGATIVE",
    },
    {
        "instruction": "Classify the sentiment of this review.",
        "input": "Exceeded my expectations — fast shipping and great quality!",
        "output": "POSITIVE",
    },
]

# Validate dataset quality
def validate_dataset(examples: list[dict]) -> None:
    required_keys = {"instruction", "input", "output"}
    for i, ex in enumerate(examples):
        missing = required_keys - set(ex.keys())
        if missing:
            raise ValueError(f"Example {i} missing keys: {missing}")
        for key, val in ex.items():
            if not val.strip():
                raise ValueError(f"Example {i} has empty '{key}'")
    print(f"Dataset valid: {len(examples)} examples, all keys present and non-empty")

validate_dataset(hf_examples)

Running a Fine-Tune

The easiest starting point is the OpenAI fine-tuning API — no GPUs to provision, pay-per-use, and the resulting model is hosted for you. For open-source models (Llama 3, Mistral, Qwen), HuggingFace + PEFT gives you full control at the cost of managing compute.

openai_finetune.py
import os, time
from openai import OpenAI

client = OpenAI(api_key=os.environ["OPENAI_API_KEY"])

# --- Step 1: Upload training data ---
with open("training_data.jsonl", "rb") as f:
    upload_response = client.files.create(file=f, purpose="fine-tune")

file_id = upload_response.id
print(f"Uploaded training file: {file_id}")

# --- Step 2: Create the fine-tuning job ---
job = client.fine_tuning.jobs.create(
    training_file=file_id,
    model="gpt-3.5-turbo",  # or "gpt-4o-mini" (check current availability)
    hyperparameters={
        "n_epochs": 3,          # 3 epochs is a good starting point
        "batch_size": "auto",   # OpenAI will choose based on dataset size
        "learning_rate_multiplier": "auto",
    },
    suffix="json-extractor",  # appended to model name for easy identification
)

job_id = job.id
print(f"Fine-tune job created: {job_id}")

# --- Step 3: Poll for completion ---
while True:
    status = client.fine_tuning.jobs.retrieve(job_id)
    print(f"Status: {status.status}")
    if status.status in ("succeeded", "failed", "cancelled"):
        break
    time.sleep(30)

if status.status == "succeeded":
    model_id = status.fine_tuned_model
    print(f"Fine-tuned model ready: {model_id}")

    # --- Step 4: Use the fine-tuned model ---
    response = client.chat.completions.create(
        model=model_id,
        messages=[
            {"role": "system", "content": "You are a JSON extraction assistant."},
            {"role": "user", "content": "Extract name and age: 'Alice Chen, 31, software engineer.'"},
        ]
    )
    print("Response:", response.choices[0].message.content)
Cost and time estimates

OpenAI GPT-3.5 fine-tuning: ~$0.008/1K training tokens. A 100-example dataset with ~500 tokens/example × 3 epochs ≈ $1.20. Training takes 10–30 minutes. HuggingFace LoRA on a free Colab T4 GPU: 200 examples trains in about 15 minutes.

Evaluating and Iterating

After training, always run your evaluation suite against both the base model and fine-tuned model before promoting to production. Key things to measure:

  • Task performance: Does the fine-tuned model score higher on your golden dataset?
  • Format compliance: Does it consistently produce the required output format?
  • Generalization: Does it perform well on inputs not seen during training?
  • Regression: Does it still handle basic tasks the base model could do?
Symptom Likely Cause Fix
Perfect on training data, poor on new inputs Overfitting — training set too small or too repetitive Add more diverse examples; reduce n_epochs
Model ignores format instructions entirely Too few training examples showing the desired format Add 2–3x more examples with varied format demonstrations
Model "forgets" basic capabilities (e.g., counting) Catastrophic forgetting due to narrow training data Add diverse "maintenance" examples covering basic tasks
Model produces the right format but wrong content Base model knowledge is insufficient; format ≠ knowledge Combine fine-tuning with RAG for the knowledge component
Iterating on fine-tuned models

Fine-tuning is rarely a one-shot process. Plan for 3–5 iterations: train → eval → analyze errors → add targeted examples → retrain. Each iteration, focus on the specific failure modes your evaluation surfaced. Keep all your training data versioned so you can reproduce any model.

Lab: Fine-tune a Small Model with LoRA
Use HuggingFace + PEFT to fine-tune GPT-2 on a custom dataset

You'll fine-tune GPT-2 (or a small Llama variant) using LoRA on a custom dataset — all on a free Colab T4 GPU. By the end you'll have a model that produces outputs in a specific format that the base model doesn't naturally use.

📓 Open in Colab
  1. Install PEFT and dependencies
    Install transformers peft datasets accelerate bitsandbytes. Load GPT-2 and its tokenizer. Verify the base model generates unstructured completions for your task.
  2. Prepare your dataset
    Create 50–100 instruction/output pairs in a specific format (e.g., JSON extraction, sentiment classification, or a custom greeting style). Format them as HuggingFace Dataset objects.
  3. Configure and apply LoRA
    Create a LoraConfig with rank 16, target the attention projection layers (c_attn for GPT-2). Apply it with get_peft_model(). Print the trainable parameter count and compare to total parameters.
  4. Train with the Trainer API
    Use HuggingFace Trainer with 3 epochs, batch size 4, and learning rate 3e-4. Monitor training loss — it should decrease steadily without going to near-zero (overfitting signal).
  5. Evaluate: base vs. fine-tuned
    Run 10 test prompts through both the base and fine-tuned model. Measure format compliance rate (did it produce the required format?). Calculate the improvement percentage.

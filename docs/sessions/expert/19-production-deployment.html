Cost Management

At prototype scale, LLM API costs feel negligible. In production with real traffic, they can become your largest infrastructure expense. A single GPT-4o call processing a 10,000-token document costs roughly $0.025 — run that 100,000 times a day and you're spending $2,500 daily, $75,000 monthly. Cost management is not optional at scale.

Understanding Token Pricing

All major LLM APIs charge separately for input tokens (what you send) and output tokens (what the model generates). Output tokens are typically 3–5× more expensive per token because generation is computationally more intensive than processing input. This means:

  • Long system prompts are costly — every request pays for the full system prompt, even if 90% is the same each time.
  • Minimize output length — instruct the model to be concise when you don't need verbose responses.
  • Choose the right model — use the smallest, cheapest model that meets your quality bar for each task.

Prompt Caching

Prompt caching is one of the highest-impact cost optimizations available. If your system prompt is 2,000 tokens and you make 100,000 requests/day, that's 200M input tokens just for the system prompt. With caching:

  • Anthropic cache_control: Mark prompt prefixes as cacheable. Cache hits cost ~10% of normal input price. Cache TTL is 5 minutes (refreshed on each cache hit).
  • OpenAI automatic caching: OpenAI automatically caches prompt prefixes longer than 1024 tokens at 50% of input price. No code changes required — just ensure your system prompt is consistent across requests.
cost_estimation.py
from dataclasses import dataclass

@dataclass
class ModelPricing:
    input_per_million: float   # USD per 1M input tokens
    output_per_million: float  # USD per 1M output tokens
    cached_input_per_million: float = 0.0  # USD per 1M cached input tokens

PRICING = {
    "gpt-4o-2024-08-06":        ModelPricing(2.50,  10.00, 1.25),
    "gpt-4o-mini-2024-07-18":   ModelPricing(0.15,   0.60, 0.075),
    "claude-3-5-sonnet-20241022": ModelPricing(3.00, 15.00, 0.30),
    "claude-3-haiku-20240307":  ModelPricing(0.25,   1.25, 0.03),
}

def estimate_monthly_cost(
    model: str,
    avg_input_tokens: int,
    avg_output_tokens: int,
    requests_per_day: int,
    cached_input_tokens: int = 0,
    cache_hit_rate: float = 0.8
) -> dict:
    """
    Estimate monthly API cost with and without prompt caching.

    Args:
        cached_input_tokens: tokens in the system prompt / shared prefix
        cache_hit_rate: fraction of requests that hit the cache (0.0-1.0)
    """
    pricing = PRICING[model]
    monthly_requests = requests_per_day * 30

    # Without caching
    base_cost = monthly_requests * (
        avg_input_tokens / 1_000_000 * pricing.input_per_million +
        avg_output_tokens / 1_000_000 * pricing.output_per_million
    )

    # With caching: reduce cost of cached prefix on cache hits
    cache_savings = (
        monthly_requests * cache_hit_rate *
        cached_input_tokens / 1_000_000 *
        (pricing.input_per_million - pricing.cached_input_per_million)
    )
    cached_cost = base_cost - cache_savings

    return {
        "model": model,
        "monthly_requests": monthly_requests,
        "base_monthly_cost_usd": round(base_cost, 2),
        "cached_monthly_cost_usd": round(cached_cost, 2),
        "monthly_savings_usd": round(cache_savings, 2),
        "cache_hit_rate": cache_hit_rate,
    }

# Example: customer support bot
result = estimate_monthly_cost(
    model="claude-3-5-sonnet-20241022",
    avg_input_tokens=3000,      # system prompt + context + user message
    avg_output_tokens=300,      # typical response length
    requests_per_day=10_000,
    cached_input_tokens=2000,   # 2000-token system prompt
    cache_hit_rate=0.85
)
print(f"Without caching: ${result['base_monthly_cost_usd']:,.2f}/month")
print(f"With caching:    ${result['cached_monthly_cost_usd']:,.2f}/month")
print(f"Monthly savings: ${result['monthly_savings_usd']:,.2f}")

Prompt Compression

For RAG applications where retrieved context is large, consider:

  • Selective retrieval: Return only the top-3 chunks instead of top-10.
  • Chunk compression: Summarize long retrieved passages before including them.
  • LLMLingua: Microsoft's open-source tool that compresses prompts by up to 20× with minimal quality loss by removing low-information tokens.

Latency Optimization

Latency is the most user-facing performance metric for AI applications. A 10-second wait with no feedback feels broken. The same 10 seconds with streaming feels fast. Understanding the sources of latency and the tools to reduce it is essential for production-quality apps.

Streaming

Streaming is the highest-leverage latency optimization and requires zero changes to your prompts or model selection. Instead of waiting for the full response, you receive tokens as they are generated and show them to the user incrementally. Time-to-first-token (TTFT) for a 500-token response might be 200ms; time-to-last-token might be 8 seconds. With streaming, the user sees output immediately.

UX Insight

Use streaming to show output as it generates — a 10s response with streaming feels much faster than a 3s response with no streaming. Users will read the beginning while the rest generates. This is table stakes for chat interfaces.

streaming_example.py
from openai import OpenAI
import time

client = OpenAI()

def stream_response(messages: list, model: str = "gpt-4o") -> str:
    """Stream a chat completion, printing tokens as they arrive."""
    start = time.perf_counter()
    first_token_time = None
    full_response = []

    with client.chat.completions.stream(
        model=model,
        messages=messages
    ) as stream:
        for event in stream:
            if event.choices and event.choices[0].delta.content:
                token = event.choices[0].delta.content
                if first_token_time is None:
                    first_token_time = time.perf_counter()
                    ttft = (first_token_time - start) * 1000
                    print(f"\n[TTFT: {ttft:.0f}ms]\n")
                print(token, end="", flush=True)
                full_response.append(token)

    total_time = (time.perf_counter() - start) * 1000
    print(f"\n\n[Total: {total_time:.0f}ms]")
    return "".join(full_response)

# FastAPI streaming endpoint example
from fastapi import FastAPI
from fastapi.responses import StreamingResponse

app = FastAPI()

@app.post("/chat")
async def chat_stream(user_message: str):
    async def generate():
        with client.chat.completions.stream(
            model="gpt-4o",
            messages=[{"role": "user", "content": user_message}]
        ) as stream:
            for event in stream:
                if event.choices and event.choices[0].delta.content:
                    # Server-Sent Events format
                    yield f"data: {event.choices[0].delta.content}\n\n"
        yield "data: [DONE]\n\n"

    return StreamingResponse(generate(), media_type="text/event-stream")

Latency Targets and Measurement

In production, track latency as a distribution, not just an average. The p99 latency (worst 1% of requests) is what causes user complaints. Common targets for interactive AI applications:

Metric Target (streaming) Target (non-streaming) Notes
Time-to-first-token (P50) < 300ms N/A What users notice most in chat
Time-to-first-token (P99) < 1000ms N/A Alert if consistently above this
Total response time (P50) N/A < 3s For classification/extraction APIs
Total response time (P99) N/A < 10s Timeout threshold for clients

Model Size Tradeoffs

Larger models are slower and more expensive, but not always better for your specific task. A well-prompted small model often matches a poorly prompted large model. Use model routing: classify the complexity of each request and route simple requests to a smaller, faster, cheaper model:

  • Simple extraction, classification, yes/no decisions: GPT-4o-mini, Claude Haiku, Gemini Flash — ~$0.15/1M input, <200ms TTFT.
  • Complex reasoning, code generation, nuanced writing: GPT-4o, Claude Sonnet — ~$2.50/1M input, 300–500ms TTFT.
  • Frontier reasoning, research-grade outputs: o1, Claude Opus — $15+/1M input, 1–5s TTFT.

Reliability & Resilience

LLM APIs are external services — they have rate limits, occasional outages, transient errors, and capacity constraints. A production AI system must be designed to handle all of these gracefully without crashing or returning errors to users.

Retry with Exponential Backoff

The most common production errors from LLM APIs are:

  • 429 Rate Limit Exceeded: Retry after a delay. Check the Retry-After header if present.
  • 500/503 Server Errors: Transient infrastructure issues. Retry with backoff.
  • 408 Timeout: Network or server timeout. Retry with a potentially shorter prompt.
resilient_client.py
import time
import logging
from openai import OpenAI, RateLimitError, APIStatusError, APITimeoutError
from tenacity import (
    retry,
    stop_after_attempt,
    wait_exponential,
    retry_if_exception_type,
    before_sleep_log,
)

logger = logging.getLogger(__name__)
client = OpenAI()

# Retry on rate limits and server errors, up to 5 attempts
# Wait: 1s, 2s, 4s, 8s between attempts (exponential backoff)
@retry(
    retry=retry_if_exception_type((RateLimitError, APIStatusError, APITimeoutError)),
    wait=wait_exponential(multiplier=1, min=1, max=60),
    stop=stop_after_attempt(5),
    before_sleep=before_sleep_log(logger, logging.WARNING),
    reraise=True,
)
def call_llm_with_retry(messages: list, model: str = "gpt-4o", **kwargs) -> str:
    """Resilient LLM call with automatic retry on transient errors."""
    response = client.chat.completions.create(
        model=model,
        messages=messages,
        timeout=30.0,  # Hard timeout per attempt
        **kwargs
    )
    return response.choices[0].message.content

# Fallback: try primary model, fall back to cheaper model on failure
PRIMARY_MODEL = "gpt-4o-2024-08-06"
FALLBACK_MODEL = "gpt-4o-mini-2024-07-18"

def call_with_fallback(messages: list, **kwargs) -> tuple[str, str]:
    """Try primary model; fall back to secondary on failure."""
    try:
        result = call_llm_with_retry(messages, model=PRIMARY_MODEL, **kwargs)
        return result, PRIMARY_MODEL
    except Exception as e:
        logger.error(f"Primary model failed: {e}. Falling back to {FALLBACK_MODEL}")
        result = call_llm_with_retry(messages, model=FALLBACK_MODEL, **kwargs)
        return result, FALLBACK_MODEL

# Circuit breaker: stop trying after sustained failures
class CircuitBreaker:
    """
    Prevent cascading failures by stopping calls to a failing service.
    States: CLOSED (normal) → OPEN (stop all calls) → HALF_OPEN (test recovery)
    """
    def __init__(self, failure_threshold: int = 5, timeout: int = 60):
        self.failure_threshold = failure_threshold
        self.timeout = timeout
        self.failures = 0
        self.last_failure_time = 0
        self.state = "CLOSED"  # CLOSED | OPEN | HALF_OPEN

    def call(self, fn, *args, **kwargs):
        if self.state == "OPEN":
            if time.time() - self.last_failure_time > self.timeout:
                self.state = "HALF_OPEN"
            else:
                raise RuntimeError("Circuit breaker OPEN — service unavailable")

        try:
            result = fn(*args, **kwargs)
            if self.state == "HALF_OPEN":
                self.state = "CLOSED"
                self.failures = 0
            return result
        except Exception as e:
            self.failures += 1
            self.last_failure_time = time.time()
            if self.failures >= self.failure_threshold:
                self.state = "OPEN"
                logger.error(f"Circuit breaker opened after {self.failures} failures")
            raise

breaker = CircuitBreaker(failure_threshold=5, timeout=60)

Observability

"You can't improve what you can't measure." In AI applications, observability means capturing enough data about every LLM interaction to understand: what questions are users asking? Are answers high quality? Where is the system slow or expensive? When does it fail?

What to Log

  • Request metadata: Timestamp, user ID, session ID, model, temperature settings.
  • Input/output content: The full prompt (sanitized of PII) and response. This is critical for debugging and quality review.
  • Token counts: Prompt tokens, completion tokens, cached tokens — needed for cost attribution.
  • Latency: Time-to-first-token, total duration, API call duration vs. application overhead.
  • Errors: Error type, status code, retry count, fallback used.
  • User feedback: Thumbs up/down, corrections, follow-up questions (signals quality).
logging_wrapper.py
import time
import uuid
import json
import logging
from functools import wraps
from openai import OpenAI
from openai.types.chat import ChatCompletion

logger = logging.getLogger(__name__)
client = OpenAI()

def log_llm_call(func):
    """Decorator that logs all LLM API calls with full telemetry."""
    @wraps(func)
    def wrapper(*args, **kwargs):
        call_id = str(uuid.uuid4())[:8]
        start = time.perf_counter()

        log_entry = {
            "call_id": call_id,
            "model": kwargs.get("model", "unknown"),
            "timestamp": time.strftime("%Y-%m-%dT%H:%M:%SZ", time.gmtime()),
            "status": "started",
        }

        try:
            result = func(*args, **kwargs)
            elapsed_ms = (time.perf_counter() - start) * 1000

            if isinstance(result, ChatCompletion):
                usage = result.usage
                log_entry.update({
                    "status": "success",
                    "latency_ms": round(elapsed_ms, 1),
                    "prompt_tokens": usage.prompt_tokens,
                    "completion_tokens": usage.completion_tokens,
                    "total_tokens": usage.total_tokens,
                    "cached_tokens": getattr(usage, "prompt_tokens_details", {}) and
                                     usage.prompt_tokens_details.cached_tokens or 0,
                    "finish_reason": result.choices[0].finish_reason,
                })
                # Estimate cost (gpt-4o pricing as example)
                cost = (usage.prompt_tokens / 1_000_000 * 2.50 +
                        usage.completion_tokens / 1_000_000 * 10.00)
                log_entry["estimated_cost_usd"] = round(cost, 6)

            logger.info(json.dumps(log_entry))
            return result

        except Exception as e:
            elapsed_ms = (time.perf_counter() - start) * 1000
            log_entry.update({
                "status": "error",
                "error_type": type(e).__name__,
                "error_message": str(e),
                "latency_ms": round(elapsed_ms, 1),
            })
            logger.error(json.dumps(log_entry))
            raise

    return wrapper

@log_llm_call
def create_completion(messages: list, model: str = "gpt-4o", **kwargs) -> ChatCompletion:
    return client.chat.completions.create(
        model=model,
        messages=messages,
        **kwargs
    )

Production Checklist

Before shipping any AI feature to real users, work through this checklist. Every item represents a category of production failure that has burned real teams.

Security

  • API keys in environment variables, never hardcoded or committed to git. Use a secrets manager (AWS Secrets Manager, HashiCorp Vault, Doppler) in production.
  • API key rotation schedule established. Rotate every 90 days or immediately if a key is suspected compromised.
  • Per-user rate limiting to prevent abuse. Even if your LLM provider rate-limits you, a malicious user could exhaust your quota for everyone else.
  • Input validation and sanitization — maximum input length, content moderation, injection detection.

Quality & Reliability

  • Model version pinned — use specific versioned identifiers, not aliases.
  • Output moderation for harmful content before returning to users.
  • Retry logic with exponential backoff for all API calls.
  • Fallback model or degraded mode defined for when primary model is unavailable.
  • Timeouts configured on all external calls. Never wait indefinitely.
  • Eval suite passing — automated evals run in CI on every prompt change.

Observability & Operations

  • Request logging enabled with latency, token counts, and costs.
  • Alerts configured on error rate, P99 latency, and daily cost spikes.
  • Runbook documented — what to do when the LLM API is down, costs spike, or quality degrades.
  • On-call rotation defined for production AI systems.
Silent Failure Risk

Never use an unpinned model in production (e.g. gpt-4o instead of gpt-4o-2024-08-06) — model updates can break your prompts silently. OpenAI updates model aliases without notice. Your carefully tuned prompts can suddenly produce different outputs on a Tuesday morning with no error, no alert, and no way to know it happened.

rate_limiting.py
"""
Per-user rate limiting using Redis sorted sets (sliding window algorithm).
Prevents any single user from consuming disproportionate API quota.
"""
import time
import redis
from fastapi import HTTPException, Request

redis_client = redis.Redis(host="localhost", port=6379, decode_responses=True)

def check_rate_limit(
    user_id: str,
    max_requests: int = 10,
    window_seconds: int = 60,
) -> None:
    """
    Sliding window rate limiter using Redis sorted sets.
    Raises HTTPException 429 if rate limit exceeded.
    """
    key = f"rate_limit:{user_id}"
    now = time.time()
    window_start = now - window_seconds

    pipe = redis_client.pipeline()
    # Remove old entries outside the window
    pipe.zremrangebyscore(key, 0, window_start)
    # Count current entries in window
    pipe.zcard(key)
    # Add current request
    pipe.zadd(key, {str(now): now})
    # Set TTL so keys auto-expire
    pipe.expire(key, window_seconds + 1)
    _, request_count, *_ = pipe.execute()

    if request_count >= max_requests:
        raise HTTPException(
            status_code=429,
            detail={
                "error": "rate_limit_exceeded",
                "message": f"Too many requests. Limit: {max_requests} per {window_seconds}s.",
                "retry_after": window_seconds,
            }
        )

# FastAPI middleware usage
from fastapi import Depends

def get_current_user(request: Request) -> str:
    """Extract user ID from JWT/session — simplified for example."""
    return request.headers.get("X-User-ID", "anonymous")

Lab: Deploy with Logging & Rate Limits
Add production-grade observability, retry logic, and rate limiting to an AI app, then deploy to Railway

Take a simple chat API endpoint and harden it for production: structured logging, retry with backoff, per-user rate limiting, and deployment to a cloud platform.

  1. Start with a bare FastAPI chat endpoint — a single POST /chat route that calls the OpenAI API and returns the response. No error handling, no logging.
  2. Add structured logging — wrap your OpenAI call with the @log_llm_call decorator from this session. Configure Python logging to output JSON to stdout. Verify you can see token counts and latency in your terminal.
  3. Add retry with tenacity — decorate your OpenAI call with the @retry decorator configured to retry on 429 and 5xx errors with exponential backoff. Test by temporarily reducing your API tier to trigger rate limits.
  4. Add Redis rate limiting — run Redis locally via Docker (docker run -p 6379:6379 redis:alpine), implement the sliding window rate limiter, and add it as a dependency to your endpoint. Test that it returns 429 after the limit.
  5. Pin your model version — change any "gpt-4o" references to "gpt-4o-2024-08-06". Add a comment explaining why versioning is important.
  6. Deploy to Railway — create a Railway project, add a Redis plugin, set your OPENAI_API_KEY environment variable, and deploy. Verify the health endpoint returns 200.
  7. Validate in production — send 25 requests in 60 seconds to your deployed endpoint and confirm the rate limiter returns 429 for the 21st+ request. Check your logs are appearing in Railway's log viewer.

19 sessions. Prompting, RAG, agents, evals, fine-tuning, deployment, responsible AI. Now it's time to build something real.

The capstone is the most important part of this course. Reading about AI is valuable. Building AI systems is transformative. The capstone exists to bridge the gap between understanding concepts and actually shipping something that works in the real world.

You have 5 project tracks to choose from — each designed to be achievable in 4 weeks of part-time effort while demonstrating the core skills you've built across the course. Choose the one that genuinely interests you. The best AI project is one you actually want to use.

What a Successful Capstone Looks Like

  • Working demo: A deployed app or script that actually runs on real inputs, not just toy examples.
  • Evaluation results: Evidence that it works — metrics, test cases, or user feedback.
  • Clean README: Someone else could set it up from your documentation.
  • Deployment link: If it's a web app, it runs somewhere other than localhost.
  • Reflection: What did you learn? What would you do differently?

Recommended Project Structure

Four weeks is enough to build something genuinely impressive if you scope it right and avoid perfection paralysis. The week-by-week structure below is proven: it ships something working by week 2 and spends the back half on quality, not catch-up.

Week 1
Scope & Design
  • Write a one-paragraph project description
  • Define 3 core features (not 10)
  • Choose your tech stack
  • Set up repo and dev environment
  • Define success criteria and eval plan
Week 2
Core Functionality
  • Build the simplest end-to-end path
  • One feature, fully working
  • Basic UI or CLI that you can demo
  • Write your first eval tests
Week 3
Evaluation & Polish
  • Add remaining core features
  • Run eval suite; fix failing cases
  • Handle edge cases and error states
  • Add logging and observability
Week 4
Deploy & Share
  • Deploy to Railway / Vercel / Fly.io
  • Write README with setup guide
  • Record a 2-minute demo video
  • Share and get feedback

Start Small, Ship Early

Start with the simplest version that demonstrates the core idea. A working simple version beats a broken complex one every time. Get the end-to-end path working first — input → LLM → output — then add complexity layer by layer.

Evaluation Requirements

Every capstone project must include an evaluation suite. This is non-negotiable — it's what separates a toy prototype from a system you'd trust in production. Your eval should:

  • Contain at least 20 test cases (input + expected output or quality criteria).
  • Run automatically with a single command (python evals/run_evals.py).
  • Report a clear score (e.g., "Passed 17/20 test cases, average quality score: 4.1/5.0").
  • Include at least 3 adversarial / edge case examples.

Deployment Requirements

Your project must be runnable by someone else. At minimum: a working README.md with setup instructions. Better: a deployed public URL. Best: a Docker container that anyone can run with docker compose up.

Recommended free/cheap deployment platforms:

  • Railway — easiest for Python backends; free tier available; Redis and Postgres plugins.
  • Fly.io — generous free tier; great for Docker containers.
  • Vercel — ideal for Next.js frontends; serverless Python with limitations.
  • Hugging Face Spaces — free for Gradio/Streamlit apps; great for ML demos.
  • Render — free tier for web services; auto-deploys from GitHub.

Lab: Project Kickoff Template
Structure your capstone project from day one

Use this structured kickoff to start your capstone project on solid footing. Completing this template before writing any code will save you days of rework. The most common reason AI projects fail is vague scope — not technical skill.

  1. Write a 1-paragraph project description — In plain English: what does it do, for whom, and why does it matter? Start with the sentence: "My project is a [type of system] that helps [target user] to [accomplish goal] by [core mechanism]." Example: "My project is a Meeting Assistant that helps remote teams by transcribing and summarizing recorded meetings using Whisper and GPT-4o, extracting action items with owners and deadlines."
  2. List your 3 core features — Not 10 nice-to-haves. The 3 features that, if working, would make someone want to use this. Write them as user stories: "As a [user], I can [action] so that [outcome]." If you have more than 3, move everything else to a "Future work" section.
  3. Define your eval criteria — Before writing any code, decide: how will you know it's working? Write at least 5 specific test cases with inputs and expected outputs. Example for Meeting Assistant: "Given a 5-minute meeting about product priorities, the system should identify at least 3 action items and correctly attribute owners mentioned by name." Specific is better than general.
  4. Set up your repo and dev environment — Create a GitHub repo. Add a .env.example file with all required environment variables listed. Set up a Python virtual environment and pin your initial dependencies in requirements.txt. Create the directory structure from the template in Section 3.
  5. Build the simplest end-to-end prototype first — Before any UI, error handling, or polish: make the core pipeline work in a Jupyter notebook or a 50-line script. Input → LLM call → output. Show it to someone. Get feedback. Then add layers.
What makes a good capstone? (Click to reveal)

A good capstone has four ingredients:

  1. A real user need — It solves a problem someone actually has. "I wanted this for myself" is perfectly valid evidence of a real need.
  2. A working demo — Not a screenshot of code. A live system you can hand to someone and watch them use it, even if it's rough around the edges.
  3. An evaluation that proves it works — Numbers. "It answers 18/20 test questions correctly" is evidence. "It seems to work pretty well" is not.
  4. A clean README — If a stranger can't set it up from your documentation in under 15 minutes, it's not done. Great documentation is a sign of a great engineer.

Model Evaluation

Intermediate 55 min Session 7 of 20

Why Accuracy Isn't Enough

Accuracy — the percentage of correct predictions — sounds like the obvious metric. But it fails completely in the presence of class imbalance, which is everywhere in real-world ML.

Consider a fraud detection model. Suppose 99% of transactions are legitimate and only 1% are fraudulent. A model that always predicts "not fraud" — without looking at a single feature — achieves 99% accuracy. It is completely useless and yet looks great by accuracy alone.

This is the core problem: accuracy doesn't tell you which cases the model gets wrong, and some errors are far more costly than others. Missing a fraud (false negative) costs money. Flagging a legitimate transaction (false positive) annoys customers. The right metric depends on which error you care more about.

Rule of Thumb: Whenever your classes are imbalanced (more than 80/20 split), treat accuracy as a starting point only. Always look at precision, recall, and F1 alongside it.

The Confusion Matrix

The confusion matrix breaks down predictions into four categories. We'll use a spam filter as our running example where the "positive" class is spam.

Predicted: Spam Predicted: Not Spam
Actual: Spam TP — True Positive
Correctly caught spam 		FN — False Negative
Spam that slipped through
Actual: Not Spam FP — False Positive
Legit email marked as spam 		TN — True Negative
Correctly passed legit email
  • True Positive (TP): Model said spam, it really was spam. Correct.
  • True Negative (TN): Model said not spam, it really wasn't. Correct.
  • False Positive (FP): Model said spam, but it was legitimate. The model cried wolf.
  • False Negative (FN): Model said not spam, but it was spam. The bad email got through.
Context Matters: False Positives and False Negatives have different costs depending on the application. In cancer screening, a False Negative (missed cancer) is catastrophic. In a spam filter, a False Positive (good email in spam folder) is merely annoying.

Precision, Recall, and F1

From the confusion matrix, we derive three metrics that tell a richer story than accuracy:

Metric Formula Answers the Question Prefer When
Precision TP / (TP + FP) Of all items flagged as positive, what fraction actually were? FP are costly — e.g., don't want to spam-block real emails
Recall TP / (TP + FN) Of all actual positives, what fraction did we catch? FN are costly — e.g., don't want to miss cancer diagnoses
F1 Score 2 × (P × R) / (P + R) Single balanced score — harmonic mean of precision and recall You need one metric that balances both concerns

The harmonic mean penalizes extreme imbalances. An F1 score of 0.9 requires both precision and recall to be high. A model with precision=1.0 and recall=0.1 gets F1=0.18 — a justifiably low score.

sklearn_metrics.py 
from sklearn.metrics import (
    classification_report,
    confusion_matrix,
    precision_score,
    recall_score,
    f1_score
)

# y_test: true labels, y_pred: model predictions
print("Confusion Matrix:")
print(confusion_matrix(y_test, y_pred))

# Full report with per-class and weighted averages
print("\nClassification Report:")
print(classification_report(y_test, y_pred, target_names=["Not Spam", "Spam"]))

# Individual metrics
precision = precision_score(y_test, y_pred)
recall    = recall_score(y_test, y_pred)
f1        = f1_score(y_test, y_pred)
print(f"Precision: {precision:.3f}")
print(f"Recall:    {recall:.3f}")
print(f"F1 Score:  {f1:.3f}")

AUC-ROC Curve

Most classifiers output a probability (e.g., 0.87 probability of spam), and you apply a threshold (e.g., 0.5) to convert that to a class label. The choice of threshold trades off precision vs recall.

The ROC (Receiver Operating Characteristic) curve shows this tradeoff at every possible threshold, plotting:

  • Y-axis: True Positive Rate = recall = TP/(TP+FN)
  • X-axis: False Positive Rate = FP/(FP+TN)

AUC (Area Under the Curve) summarizes this as a single number:

  • AUC = 1.0: Perfect model at all thresholds
  • AUC = 0.5: Random guessing (diagonal line)
  • AUC < 0.5: Worse than random — model has inverted its predictions
roc_auc.py 
from sklearn.metrics import roc_auc_score, roc_curve
import matplotlib.pyplot as plt

# Use predict_proba to get probability scores (not just 0/1 labels)
y_proba = model.predict_proba(X_test)[:, 1]   # probability of positive class

auc = roc_auc_score(y_test, y_proba)
print(f"AUC-ROC: {auc:.3f}")

# Plot the curve
fpr, tpr, thresholds = roc_curve(y_test, y_proba)
plt.figure(figsize=(6, 5))
plt.plot(fpr, tpr, label=f"Model (AUC = {auc:.2f})")
plt.plot([0, 1], [0, 1], "k--", label="Random (AUC = 0.5)")
plt.xlabel("False Positive Rate")
plt.ylabel("True Positive Rate (Recall)")
plt.title("ROC Curve")
plt.legend()
plt.tight_layout()
plt.show()
When to Use AUC-ROC: Use AUC-ROC when you need a threshold-independent metric — for example, when comparing different models before deciding on a deployment threshold. For heavily imbalanced datasets, Precision-Recall AUC is often more informative.

Cross-Validation

A single train/test split is fragile. If you happen to split such that all the hard examples land in training and easy ones in test, you'll overestimate performance — and vice versa. K-fold cross-validation fixes this by using every sample for both training and validation.

How it works with K=5 folds:

  1. Split data into 5 equal chunks (folds).
  2. Train on folds 1–4, validate on fold 5. Record score.
  3. Train on folds 1–3 and 5, validate on fold 4. Record score.
  4. Repeat until every fold has been the validation set once.
  5. Average the 5 scores. This average is your cross-validated performance estimate.
cross_validation.py 
from sklearn.model_selection import cross_val_score, StratifiedKFold
from sklearn.ensemble import RandomForestClassifier
import numpy as np

model = RandomForestClassifier(n_estimators=100, random_state=42)

# StratifiedKFold preserves class proportions in each fold
cv = StratifiedKFold(n_splits=5, shuffle=True, random_state=42)

scores = cross_val_score(
    model, X, y,
    cv=cv,
    scoring="f1"      # or "accuracy", "roc_auc", "precision", "recall"
)

print(f"F1 per fold:  {scores.round(3)}")
print(f"Mean F1:      {scores.mean():.3f}")
print(f"Std deviation: {scores.std():.3f}")
# Low std = consistent performance across different data subsets
# High std = model is sensitive to which data it sees — investigate further

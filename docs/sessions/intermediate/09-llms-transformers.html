Tokens: The Language of LLMs

LLMs don't read words or characters — they process tokens. A tokenizer splits text into chunks (tokens) and converts each chunk to an integer ID. The model sees a sequence of these IDs, processes them, and predicts the next one.

Tokens are not words. In English, most common words are one token, but longer or less common words split into multiple tokens. Spaces, punctuation, and special characters each get their own tokens.

Text Tokens (approximate) Token Count
"Hello, world!" ["Hello", ",", " world", "!"] 4 tokens
"Backpropagation" ["Back", "prop", "agation"] 3 tokens
"I am happy" ["I", " am", " happy"] 3 tokens
One page (~500 words) — ~750 tokens
tokenization.py 
import tiktoken

# tiktoken is OpenAI's tokenizer library
enc = tiktoken.encoding_for_model("gpt-4o")

text = "Backpropagation is just the chain rule applied repeatedly."
tokens = enc.encode(text)
token_count = len(tokens)

print(f"Text: {text}")
print(f"Token IDs: {tokens}")
print(f"Token count: {token_count}")

# Decode back to see what each token represents
for token_id in tokens:
    print(f"  {token_id:6d} → '{enc.decode([token_id])}'")

# Count tokens for a long document
with open("document.txt") as f:
    doc = f.read()
print(f"\nDocument tokens: {len(enc.encode(doc)):,}")
print(f"Estimated pages: {len(enc.encode(doc)) / 750:.1f}")
Context Window: GPT-4 Turbo has a 128,000 token context window — roughly 100,000 words or about 200 pages of a novel. Claude 3.5 has a 200K token context. This is the maximum amount of text the model can "see" in one conversation.

Word Embeddings

Before a transformer processes tokens, each token ID is converted to a dense vector — a list of hundreds of floating-point numbers. This is an embedding. Similar words end up with similar vectors because they appear in similar contexts during training.

The classic demonstration of this geometry: if you take the vector for "king", subtract "man", and add "woman", you land very close to the vector for "queen". The model has implicitly learned gender as a direction in embedding space.

Word2Vec (2013) first demonstrated this. Modern LLMs learn vastly richer embeddings — not just word-level, but embeddings that encode syntactic roles, semantic relationships, and even factual associations.

Concept Embedding Behavior
Synonyms (happy, joyful) Similar vectors — close together in embedding space
Antonyms (hot, cold) Related but opposite — nearby in some dimensions, opposite in others
Analogies (Paris:France :: Berlin:?) Vector(Paris) - Vector(France) + Vector(Germany) ≈ Vector(Berlin)
Polysemy (bank: river vs. financial) Context-dependent embeddings (transformers handle this; Word2Vec cannot)

The Attention Mechanism

Attention is the core innovation of the transformer. At each layer, every token can attend to — look at and be influenced by — every other token in the context. This is fundamentally different from RNNs, which processed tokens one at a time and struggled to remember context from far away.

For each token, the attention mechanism computes three vectors:

  • Query (Q): "What am I looking for?" — this token's question about context.
  • Key (K): "What information do I have?" — what this token offers to others.
  • Value (V): "What I actually contain." — the content to contribute if attended to.

Attention weights are computed by comparing queries against keys (dot product, then softmax). These weights determine how much of each token's value to include in the output. Tokens with high matching scores between their key and another token's query get attended to heavily.

Why Attention Works: Attention allows the model to understand context-dependent meaning. The word "bank" attends to "river" in one sentence and "account" in another, producing different representations each time — something impossible with static word vectors.
attention_intuition.py 
import torch
import torch.nn.functional as F

# Minimal self-attention implementation (single head, for intuition)
def self_attention(Q, K, V):
    """
    Q, K, V: tensors of shape (seq_len, d_k)
    Returns: attended values, attention weights
    """
    d_k = Q.shape[-1]

    # Compute attention scores: how much does each query match each key?
    scores = Q @ K.T / (d_k ** 0.5)   # scale to prevent vanishing gradients

    # Softmax to get a probability distribution over tokens
    weights = F.softmax(scores, dim=-1)

    # Weighted sum of values
    output = weights @ V
    return output, weights

# Example: 4 tokens, each embedded as 8-dim vector
seq_len, d_k = 4, 8
Q = torch.randn(seq_len, d_k)
K = torch.randn(seq_len, d_k)
V = torch.randn(seq_len, d_k)

output, attn_weights = self_attention(Q, K, V)
print("Attention weights (each row sums to 1):")
print(attn_weights.round(decimals=2))

The Transformer Architecture

The original transformer (Vaswani et al., "Attention Is All You Need", 2017) had an encoder and decoder. Modern LLMs use variations of this.

Architecture Components Training Objective Best For Examples
Encoder-only Encoder stack Masked language modeling (fill in blanks) Understanding, classification, embeddings BERT, RoBERTa, DistilBERT
Decoder-only Decoder stack with causal masking Next-token prediction (autoregressive) Text generation, chat, code GPT-4, Claude, LLaMA, Mistral
Encoder-Decoder Both encoder and decoder Sequence-to-sequence (translate, summarize) Translation, summarization T5, BART, mT5

Pre-training trains the model on massive text datasets (essentially the internet) to learn general language patterns. This is done once and is extremely expensive (millions of dollars in compute).

Fine-tuning takes a pre-trained model and continues training on a smaller, task-specific dataset. This is far cheaper and is how companies adapt base models for specific use cases.

RLHF (Reinforcement Learning from Human Feedback) is the additional step that makes a language model into a helpful assistant — humans rate model outputs and the model is trained to produce higher-rated responses.

Hallucinations & Context Windows

LLMs don't "know" facts the way a database does. They predict the most statistically likely continuation of a sequence of tokens, based on patterns in their training data. This creates a critical limitation: the model may generate text that sounds authoritative but is factually wrong.

This is called hallucination, and it has a clear cause:

  • The training objective is to predict the next token — not to verify factual accuracy.
  • The model has no uncertainty detector — it produces confident-sounding text regardless of whether the facts are real.
  • For low-frequency facts (obscure events, specific numbers, citations), the model often generates plausible-sounding but fabricated details.

Context window limitations also affect reliability:

  • Compute cost scales roughly quadratically with context length.
  • Research shows the "lost in the middle" effect: models attend strongly to the beginning and end of long contexts but may underweight information in the middle.
  • For most tasks, shorter, focused context outperforms dumping everything in.

RAG (Retrieval-Augmented Generation) directly addresses the knowledge gap: instead of relying on the model's parametric memory, you retrieve relevant documents at query time and inject them into the context. The model then reasons over fresh, accurate information you provide. We cover RAG in depth in Sessions 10–11.

Practical Implication: Never rely on an LLM for specific facts, dates, statistics, or citations without verification. Use RAG or tool use to ground the model in a reliable knowledge source when factual accuracy matters.

RAG Pipelines

Intermediate 65 min Session 11 of 20

The Problem RAG Solves

Large language models are trained on data up to a knowledge cutoff date. They also have no idea your internal documents exist — your company's Confluence wiki, Slack history, product specs, and legal contracts are invisible to the base model. When users ask about these, the model either refuses or, worse, invents plausible-sounding but false answers.

Retrieval-Augmented Generation (RAG) fixes this by giving the model a set of relevant documents at query time. Instead of relying purely on trained weights, the model reads the retrieved context and bases its answer on it. This grounds the model in real, up-to-date, private information without the cost of fine-tuning.

Why RAG beats fine-tuning for knowledge

Without RAG, asking an LLM about your internal docs is like asking someone who's never read them. Fine-tuning can help with style and format, but it doesn't reliably store factual knowledge — and the knowledge goes stale the moment your docs update. RAG keeps knowledge current at zero retraining cost.

RAG Pipeline: 4 Stages

A RAG pipeline has a clear separation between offline indexing (done once, or on document update) and online retrieval (done per query). Here's the complete flow:

1. Chunk
→
2. Embed
→
3. Store
→
4. Retrieve + Generate

Stages 1–3: offline indexing  |  Stage 4: online (per query)

Stage What Happens Phase
1. Chunk Split documents into overlapping segments of 200–1000 tokens each Offline
2. Embed Convert each chunk to a dense vector using an embedding model Offline
3. Store Persist vectors + metadata in a vector database (Chroma, Pinecone, etc.) Offline
4. Retrieve + Generate Embed the query → find top-k chunks → inject into LLM prompt → generate answer Online

Chunking Strategies

How you split documents dramatically affects retrieval quality. Too small and you lose context ("Paris" without the surrounding sentence about why it's mentioned). Too large and you exceed context limits and dilute relevance — the retrieved chunk contains your answer buried inside unrelated content.

Strategy How It Works Best For
Fixed-size with overlap Split every N characters/tokens, overlap by M characters to preserve context at boundaries General purpose; easy to implement
Sentence-boundary aware Split on sentence endings, accumulate until a token limit is reached Prose documents, articles
Markdown/HTML header Split on heading tags — each section becomes a chunk, preserving logical units Docs, wikis, structured content
chunking.py
from langchain.text_splitter import RecursiveCharacterTextSplitter

text = """
Retrieval-Augmented Generation (RAG) is a technique that enhances LLM outputs
by fetching relevant documents at inference time. It was introduced in 2020 by
Lewis et al. at Facebook AI Research.

The core idea is simple: before generating a response, retrieve context from
an external knowledge base and inject it into the prompt. This grounds the
model in verified facts rather than relying solely on trained weights.

RAG pipelines consist of two phases: offline indexing and online retrieval.
During indexing, documents are chunked, embedded, and stored in a vector DB.
During retrieval, the query is embedded and the most similar chunks are fetched.
"""

splitter = RecursiveCharacterTextSplitter(
    chunk_size=200,        # characters per chunk (tune for your token budget)
    chunk_overlap=40,      # overlap to preserve boundary context
    separators=["\n\n", "\n", ". ", " ", ""],  # try splitting on paragraphs first
)

chunks = splitter.split_text(text)
for i, chunk in enumerate(chunks):
    print(f"--- Chunk {i+1} ({len(chunk)} chars) ---")
    print(chunk)
    print()
Choosing chunk size

A good starting point is 512 tokens with 50-token overlap for most documents. Evaluate retrieval quality by checking whether the correct chunk is in your top-k results for a sample of test queries — then adjust chunk size accordingly.

The Retrieval Step

At query time, the user's question is passed through the same embedding model used during indexing. The resulting query vector is compared against all stored chunk vectors, and the top-k most similar chunks are returned.

The basic flow:

  1. Embed the query using the same model as the documents (critical — different models produce incompatible vector spaces).
  2. Run approximate nearest neighbor (ANN) search in the vector database to get top-k chunks.
  3. Optionally re-rank: pass the query + each candidate chunk through a cross-encoder model that scores relevance more accurately (but more slowly).
Retrieval Method Speed Accuracy Use When
ANN (bi-encoder) Very fast (ms) Good Default — scales to millions of docs
Cross-encoder re-ranker Slow (100–500ms) Excellent Re-rank top-20 candidates for precision
Hybrid (BM25 + vectors) Medium Very good When exact keyword matches matter (e.g., product codes)
Add a Re-Ranker

Add a re-ranker (cross-encoder) after retrieval to improve relevance — it's slow but accurate. Retrieve top-20 with ANN, re-rank to top-5 with the cross-encoder, then pass those 5 to the LLM. This two-stage approach gives you both speed and accuracy.

Building the Augmented Prompt

The retrieved chunks are injected into the LLM's system or user prompt as context. The key design decision is where and how to inject them. Here's a minimal but production-ready RAG implementation without LangChain — just OpenAI and Chroma.

minimal_rag.py
import os
import chromadb
from openai import OpenAI
from sentence_transformers import SentenceTransformer

client = OpenAI(api_key=os.environ["OPENAI_API_KEY"])
embedder = SentenceTransformer("all-MiniLM-L6-v2")

# ----- OFFLINE: index documents -----
docs = [
    "Our refund policy allows returns within 30 days of purchase.",
    "Premium members get free shipping on all orders over $50.",
    "Customer support is available Monday–Friday, 9am–6pm EST.",
    "We ship internationally to over 80 countries.",
]
chroma = chromadb.Client()
collection = chroma.create_collection("faq")
collection.add(
    documents=docs,
    embeddings=embedder.encode(docs).tolist(),
    ids=[f"d{i}" for i in range(len(docs))],
)

# ----- ONLINE: handle a user query -----
def rag_answer(user_question: str, top_k: int = 3) -> str:
    # 1. Embed the query
    q_embedding = embedder.encode([user_question]).tolist()

    # 2. Retrieve top-k chunks
    results = collection.query(query_embeddings=q_embedding, n_results=top_k)
    context_chunks = results["documents"][0]
    context = "\n".join(f"- {c}" for c in context_chunks)

    # 3. Build the augmented prompt
    system_prompt = f"""You are a helpful customer support assistant.
Answer the user's question using ONLY the context below.
If the answer isn't in the context, say "I don't have that information."

CONTEXT:
{context}"""

    # 4. Call the LLM
    response = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[
            {"role": "system", "content": system_prompt},
            {"role": "user",   "content": user_question},
        ]
    )
    return response.choices[0].message.content

# Test it
print(rag_answer("Can I return something I bought last week?"))
print(rag_answer("Do you deliver to Japan?"))
Prompt template tip

Always instruct the model to answer only from the provided context and to explicitly say "I don't know" when the context doesn't contain the answer. Without this instruction, the model will fall back to its training knowledge and potentially hallucinate.

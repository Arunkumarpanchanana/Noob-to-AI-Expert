RAG: Retrieval-Augmented Generation

The #1 production AI pattern. Ground your LLM in real documents — eliminate hallucination, stay current, build enterprise knowledge bots.

90 min Intermediate Colab Lab 5 Quiz Questions

Why RAG Exists

LLMs have two fundamental problems for enterprise use: they hallucinate, and their knowledge is frozen at training cutoff. RAG solves both by giving the model fresh, real documents at inference time.

❌

Without RAG

"What is our refund policy?" → LLM guesses or hallucinates a policy that doesn't exist. Knowledge cutoff makes it unable to answer about recent events.

✅

With RAG

"What is our refund policy?" → System retrieves your actual policy document → LLM reads it and answers accurately with citations. Always current, never hallucinated.

The RAG Pipeline: 6 Steps

  1. Load documents — PDFs, markdown files, web pages, databases. Parse text from raw files.
  2. Chunk — Split documents into smaller pieces (e.g., 500 tokens with 50-token overlap). Smaller chunks = more precise retrieval.
  3. Embed chunks — Convert each chunk to a vector using an embedding model.
  4. Store in vector DB — Save chunk text + embedding + metadata to ChromaDB, Pinecone, etc.
  5. Retrieve — At query time: embed the query, find top-k most similar chunks.
  6. Generate — Inject retrieved chunks into the LLM prompt. Model answers using only provided context.
import chromadb
from openai import OpenAI
import PyPDF2

client = OpenAI()
chroma = chromadb.Client()

# ========== INDEXING PHASE (run once) ==========

def load_pdf(path: str) -> str:
    with open(path, "rb") as f:
        reader = PyPDF2.PdfReader(f)
        return "\n".join(page.extract_text() for page in reader.pages)

def chunk_text(text: str, chunk_size: int = 500, overlap: int = 50) -> list[str]:
    words = text.split()
    chunks = []
    for i in range(0, len(words), chunk_size - overlap):
        chunk = " ".join(words[i:i + chunk_size])
        if chunk:
            chunks.append(chunk)
    return chunks

def embed_texts(texts: list[str]) -> list[list[float]]:
    response = client.embeddings.create(model="text-embedding-3-small", input=texts)
    return [d.embedding for d in response.data]

# Load and index
document_text = load_pdf("company_handbook.pdf")
chunks = chunk_text(document_text)
embeddings = embed_texts(chunks)

collection = chroma.create_collection("company_docs")
collection.add(
    documents=chunks,
    embeddings=embeddings,
    ids=[f"chunk_{i}" for i in range(len(chunks))],
    metadatas=[{"source": "company_handbook.pdf", "chunk": i} for i in range(len(chunks))]
)

print(f"Indexed {len(chunks)} chunks")

# ========== RETRIEVAL + GENERATION (at query time) ==========

def rag_answer(question: str, top_k: int = 5) -> str:
    # Step 1: Retrieve relevant chunks
    results = collection.query(
        query_texts=[question],
        n_results=top_k
    )
    context_chunks = results["documents"][0]
    context = "\n\n---\n\n".join(context_chunks)

    # Step 2: Generate answer using retrieved context
    prompt = f"""Answer the question using ONLY the provided context.
If the answer isn't in the context, say "I don't have that information."

Context:
{context}

Question: {question}

Answer:"""

    response = client.chat.completions.create(
        model="gpt-4o",
        messages=[{"role": "user", "content": prompt}],
        temperature=0  # Low temp for factual answers
    )
    return response.choices[0].message.content

# Test
answer = rag_answer("What is our parental leave policy?")
print(answer)

Chunking Strategies

How you chunk documents dramatically affects RAG quality. Bad chunking is the #1 cause of poor RAG performance.

StrategyMethodBest For
Fixed-sizeSplit every N tokens, with overlapSimple, works for most docs. Fast to implement.
RecursiveTry to split on paragraphs → sentences → wordsBetter structure preservation. LangChain default.
SemanticSplit at topic boundaries using embedding similarityBest quality. Expensive. Groups related content.
Document-awareSplit at markdown headers, PDF sections, HTML tagsStructured docs (wikis, documentation, reports).
⚠️

Overlap Is Critical

Without overlap, a key sentence at the boundary between two chunks gets split. A 50–100 token overlap ensures context sentences aren't lost. The tradeoff: more storage and slightly more redundancy in results.

Improving RAG Quality

🔀

Hybrid Search

Combine keyword (BM25) + semantic search. Keyword catches exact matches. Semantic catches paraphrases. Merge with RRF (Reciprocal Rank Fusion).

🔄

Re-ranking

After retrieving top-20 chunks, use a cross-encoder model to re-rank and keep top-5. Cross-encoders are slower but much more accurate than bi-encoders.

❓

HyDE

Hypothetical Document Embeddings — generate a hypothetical answer, embed it, search with that embedding. Outperforms query embeddings alone.

📝

Query Expansion

Use an LLM to rewrite the query into 3 variations, search with all three, merge results. Handles vague or ambiguous queries.

Lab: Build a Chatbot Over Your Own Documents

Estimated time: 60 minutes · Google Colab

Open in Google Colab
1

Load Your Documents

Upload 3–5 PDF or markdown files to Colab. Use PyPDF2 or the python-docx library to extract text. Try Wikipedia articles if you don't have your own docs.

2

Chunk and Embed

Implement the chunking function with 500-token chunks and 50-token overlap. Embed all chunks with OpenAI's embedding model. This may take a minute for large documents.

3

Build the RAG Loop

Implement the full retrieve → generate pipeline. Ask 5 questions that require information from your documents. Verify the answers are accurate.

4

Test Hallucination Defense

Ask a question whose answer is NOT in your documents. Does the model correctly say "I don't have that information"? If not, strengthen the prompt instruction.

🏆

Stretch: Add Source Citations

Store the source filename and page number in ChromaDB metadata. Include it in the response: "According to [company_handbook.pdf, page 12]..."

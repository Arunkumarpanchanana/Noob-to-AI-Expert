Fine-Tuning

When prompting isn't enough. Learn when fine-tuning beats RAG, how to prepare training data, and how to fine-tune open-source models with LoRA.

90 min Expert Colab Lab 5 Quiz Questions

Prompting vs RAG vs Fine-Tuning

These are not competing approaches — they solve different problems. Use the decision tree below before reaching for fine-tuning.

Does the model know enough about your domain?
  → Yes → Use Prompting (fastest, cheapest)
  → No: Is the knowledge static or changes rarely?
     → Yes → Use RAG (ground model in your docs)
     → No: Do you need a specific style, format, or behavior?
        → Yes → Use Fine-Tuning (bake in style/behavior)
        → No → Start with RAG + Prompting first
⚠️

Fine-Tuning Gotchas

Fine-tuning doesn't add new knowledge reliably — use RAG for that. Fine-tuning is great for: style consistency, output format, domain-specific language patterns, and behavior that's hard to specify in a prompt. Budget $50–500 for a serious fine-tuning run.

Preparing Training Data

Quality beats quantity. 500 perfect training examples beat 5,000 mediocre ones. Use the JSONL chat format required by most fine-tuning APIs.

import json
import random

# Each training example: a conversation the model should learn from
def create_training_example(user_msg: str, assistant_reply: str, system: str = "") -> dict:
    messages = []
    if system:
        messages.append({"role": "system", "content": system})
    messages.append({"role": "user", "content": user_msg})
    messages.append({"role": "assistant", "content": assistant_reply})
    return {"messages": messages}

training_data = [
    create_training_example(
        user_msg="Summarize this in one sentence: [article about climate change...]",
        assistant_reply="Rising global temperatures driven by carbon emissions are causing extreme weather, sea level rise, and biodiversity loss at an unprecedented rate.",
        system="You are a concise summarizer. Always respond in exactly one sentence."
    ),
    create_training_example(
        user_msg="What's the capital of France?",
        assistant_reply="Paris.",
        system="You are a concise summarizer. Always respond in exactly one sentence."
    ),
]

# Split into train/validation (90/10)
random.shuffle(training_data)
split = int(len(training_data) * 0.9)
train, val = training_data[:split], training_data[split:]

# Save as JSONL
def save_jsonl(data: list, path: str):
    with open(path, "w") as f:
        for item in data:
            f.write(json.dumps(item) + "\n")

save_jsonl(train, "train.jsonl")
save_jsonl(val, "val.jsonl")
print(f"Train: {len(train)}, Val: {len(val)}")

LoRA Fine-Tuning with Hugging Face

LoRA (Low-Rank Adaptation) fine-tunes only a tiny fraction of parameters — typically 0.1–1% — making it feasible on a single GPU. This is the standard approach for open-source model fine-tuning.

# pip install transformers peft datasets accelerate bitsandbytes
# Run this in Colab with a T4 GPU runtime

from transformers import AutoTokenizer, AutoModelForCausalLM, TrainingArguments
from peft import LoraConfig, get_peft_model, TaskType
from datasets import load_dataset
import torch

MODEL_NAME = "meta-llama/Llama-3.2-1B-Instruct"  # Small model, runs on free Colab

# Load model in 4-bit quantization to fit in GPU memory
from transformers import BitsAndBytesConfig

bnb_config = BitsAndBytesConfig(
    load_in_4bit=True,
    bnb_4bit_quant_type="nf4",
    bnb_4bit_compute_dtype=torch.float16
)

model = AutoModelForCausalLM.from_pretrained(
    MODEL_NAME,
    quantization_config=bnb_config,
    device_map="auto"
)
tokenizer = AutoTokenizer.from_pretrained(MODEL_NAME)
tokenizer.pad_token = tokenizer.eos_token

# Configure LoRA — only trains a small set of adapter weights
lora_config = LoraConfig(
    task_type=TaskType.CAUSAL_LM,
    r=16,               # Rank of the update matrices (higher = more params)
    lora_alpha=32,      # Scaling factor
    lora_dropout=0.1,
    target_modules=["q_proj", "v_proj"]  # Which layers to adapt
)

model = get_peft_model(model, lora_config)
model.print_trainable_parameters()
# trainable params: 1,572,864 || all params: 1,236,408,320 || trainable%: 0.13%

When NOT to Fine-Tune

SituationBetter ApproachWhy
Need up-to-date knowledgeRAGFine-tuning bakes in static knowledge; RAG retrieves fresh docs
Need to follow complex instructionsBetter system promptInstruction following improves with prompt engineering, not fine-tuning
Have fewer than 100 examplesFew-shot promptingFine-tuning needs 500+ high-quality examples to reliably improve
Need to cite sourcesRAGFine-tuned models hallucinate citations; RAG can return actual source text
